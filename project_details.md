# The Tribunal — Project Report Analysis

## 1. PROJECT OVERVIEW

### Purpose & Problem
"The Tribunal" is a narrative-driven moral judgment web application. The player assumes the role of a judge presiding over **Elias Voss**, a warehouse worker whose daughter Maya has a chronic illness repeatedly denied insurance coverage. Across **10 sequential legal cases**, the player renders verdicts (guilty/not guilty + severity 1–10) that cumulatively shape Elias's life trajectory and simultaneously reveal the player's own decision-making patterns.

The core problem it solves is **behavioural profiling through moral decision-making**: it silently tracks the player's judicial tendencies across 4 dimensions (framework, leniency, empathy, consistency) and ultimately classifies them into one of 5 psychological archetypes. Cases 9–10 are **dynamically generated by an LLM** (Llama 3.1 70B via OpenRouter) specifically tailored to exploit the player's detected tendencies. Two endings are then generated: one for Elias's fate and one for the player's psychological state.

### Target User / Use Case
- **Primary**: Students and general users engaging with an AI-powered interactive fiction / moral reasoning experiment.
- **Educational**: Designed as a student AI course project demonstrating behavioural profiling, expert systems, and LLM-driven dynamic content generation (see `README.md:395-401`).
- **Languages**: English and Bengali (বাংলা).

## 2. AI / EXPERT SYSTEM COMPONENTS

### 2A. Forward-Chaining Expert System (Rule Engine)
**Files**: `src/expert/knowledgeBase.js`, `src/expert/inferenceEngine.js`, `src/expert/explanation.js`, `src/expert/index.js`

This is the **core AI/expert system** of the project — a hand-built **production-rule forward-chaining inference engine** with an explanation subsystem.

#### Knowledge Base (`src/expert/knowledgeBase.js`, lines 1–699)
- Contains **~40 IF-THEN production rules** organized into **5 categories**:
  - **Sentencing Rules** (14 rules, lines 2–214): Recommend severity ranges based on player profile. Example: `sent_strict_max` fires when `profile.leniency_score > 0.7` AND `profile.framework_score > 0.6` → concludes severity range `[7, 10]`.
  - **Outcome Rules** (9 rules, lines 216–372): Determine Elias's final fate tag (imprisonment, rehabilitation, exile, death, disappearance, transformation, tragedy). Each carries a confidence value (0.5–0.95).
  - **Archetype Rules** (6 rules, lines 374–448): Classify the player's psychological archetype (ironRuleKeeper, empatheticOverride, principledPragmatist, contextualJudge, strictConsequentialist). Includes a fallback rule.
  - **Escalation Rules** (14 rules, lines 450–622): Select which pre-written scenario to present next from the scenario bank, based on the player's evolving profile. Determines the difficulty tier (bank index range [0–9]).
  - **Feeling Rules** (6 rules, lines 624–698): Determine the judge's reflective emotional state for the ending (numb_questioning, guilty_peace, haunted_confused, detached_resolution, weary_wisdom).

- Each rule has: `id`, `description`, `priority` (1–10), `category`, `conditions` (field-path → operator → value), `conclusion`, and `explanation`.
- Conditions support operators: `>`, `>=`, `<`, `<=`, `==`, `!=`, `in`, `between`.
- Rule fields reference dot-paths into working memory: `profile.framework_score`, `verdictStats.guiltyRate`, `currentCase.crime_tag`, etc.

#### Inference Engine (`src/expert/inferenceEngine.js`, lines 1–181)

**Algorithm**: **Forward Chaining** with priority-based conflict resolution.

- **`forwardChain(initialWorkingMemory, ruleBase)`** (line 75):
  1. Deep-clones working memory; initializes empty conclusions and trace.
  2. Iterates up to `MAX_ITERATIONS = 100` (line 83).
  3. Each iteration: filters rules that haven't fired AND whose conditions match current working memory, sorts by priority (descending), fires the highest-priority rule.
  4. Applies the conclusion to working memory (appends severity range, outcome tag with confidence, feeling tag with confidence, archetype, or bank index range).
  5. Records a trace entry for explanation.
  6. Repeats until no new rules fire (fixpoint).

- **`buildWorkingMemory(profile, verdicts, currentCase)`** (line 146): Constructs the working memory from the player's profile scores, aggregated verdict statistics (guiltyRate, avgSeverity, etc.), and current case data.

- **`pickBestConclusions(raw)`** (line 116): Resolves conflicts — for outcome and feeling tags (which accumulate with confidence), picks the one with highest confidence. For severity and bank index, takes the last-set range.

**Inference type**: Forward chaining (data-driven). Rules fire based on current state; no backward chaining or goal-directed reasoning.

#### Explanation Subsystem (`src/expert/explanation.js`, lines 1–25)
- **`formatTrace(trace)`**: Structures trace entries for display.
- **`summarizeTrace(trace)`**: Groups trace entries by category.
- **`traceToSentences(trace)`**: Converts trace into human-readable sentences.
- The full expert trace is shown to the player in the ending screen via `ExpertTraceSection` component (`src/components/EndingScreen.jsx`, lines 64–108).

### 2B. Behavioural Profile Tracker (Numeric Scoring)
**File**: `src/hooks/useProfileTracker.js`, lines 1–63

- Tracks 4 continuous scores in `[0, 1]`: `framework_score`, `leniency_score`, `empathy_score`, `consistency_score`.
- Initialized at 0.5 each (neutral).
- **`updateProfile(profile, signalWeights, verdict)`** (line 21): After each verdict, adjusts scores by `signalWeights[field] * factor`, where `factor = verdictFactor(verdict, severity)` returns -0.6 for not-guilty, or `(severity - 5.5) / 10` for guilty (ranges from -0.45 to +0.45).
- **`updateConsistency(profile, severities)`** (line 42): Computes variance of severity scores across all guilty verdicts, normalizes, and sets `consistency_score = 1 - normalizedVariance`. High consistency = low variance.
- Signal weights are embedded in each case scenario (e.g., Case 1 at `src/data/case1.js:12-16` has `framework: -0.2, leniency: 0.25, empathy: 0.25`).

### 2C. LLM-Driven Dynamic Content Generation
**Files**: `src/api/ai.js`, `src/api/generateFinalCases.js`, `src/api/generateEndings.js`

- Uses **OpenRouter API** with model `openrouter/free` (which resolves to available free models, nominally Llama 3.1 70B — see `README.md:36`, `src/api/ai.js:2`).
- **Case Generation** (`generateFinalCases.js`): After Case 8, sends the full 8-case history + player profile to the LLM. Prompted to generate 2 morally complex cases specifically designed to exploit the player's detected tendencies.
- **Ending Generation** (`generateEndings.js`): After Case 10, sends full 10-case history + final profile + expert system conclusions (outcome_tag, feeling_tag, archetype). The LLM writes the prose endings but the **structural outcome is predetermined by the expert system** — the LLM only fills in the narrative.
- **Retry logic** (`ai.js:193-216`): Up to 3 attempts with exponential backoff (0s, 2s, 4s delays).
- **JSON extraction** (`ai.js:7-98`): Robust parsing with multiple fallback strategies — outermost brace extraction, smart-quote replacement, case-block extraction — to handle malformed LLM output.
- **Schema validation** (`src/api/schemas.js`): Uses **Zod** for strict validation of LLM responses, with tolerant fallback schemas that supply defaults for missing fields.

### 2D. Scenario Selection (Adaptive Difficulty)
**Files**: `src/App.jsx:239-261`, `src/data/scenarioBanks.js`, `src/data/scenarioBanksBN.js`

- Cases 1–8 are pre-written. Case 1 is fixed (`src/data/case1.js`).
- Cases 2–8 are selected from **scenario banks**: 10 scenarios per case number × 7 case numbers = **70 pre-written scenarios per language** (1,104 lines each).
- The expert system's **escalation rules** determine a bank index range based on the player's profile. The selection picks the midpoint index within that range (`src/App.jsx:243-256`).
- This creates adaptive difficulty without any LLM call.

7. **Archetype classification**: One of 5 archetypes with description
8. **Expert system trace**: Expandable list of all fired rules with explanations (`src/components/EndingScreen.jsx:64-108`)

### Sample Input/Output
**Input**: Case 1 — "The first offence" (Elias stealing painkillers for his daughter)
- Player selects: **Guilty**, severity **4**, reasoning: "He broke the law, but for understandable reasons"

**Processing** (via `useProfileTracker.updateProfile`):
- `signal_weights = { framework: -0.2, leniency: 0.25, empathy: 0.25 }`
- `factor = (4 - 5.5) / 10 = -0.15`
- New scores: framework = 0.5 + (-0.2 × -0.15) = 0.53, leniency = 0.5 + (0.25 × -0.15) = 0.4625, empathy = 0.5 + (0.25 × -0.15) = 0.4625

**Output**: TransitionScreen shows Elias's reaction: *"Elias lowered his eyes as the gavel fell, his shoulders folding inward as though the sentence had already been written."*

## 7. TESTING / EVALUATION

### Automated Tests
**None.** The README explicitly states: *"Currently no automated tests."* (`README.md:301`). There are no test files, no test framework dependencies, and no test scripts in `package.json`.

### Manual Testing Recommendations (from `README.md:302-305`)
- Both languages (English and Bengali)
- Both arcs (lenient vs. harsh verdict patterns)
- Edge cases (all guilty vs. all not guilty)

### Quality Assurance Measures in Code
- **Zod schema validation** (`src/api/schemas.js`): Strict + tolerant schemas for all LLM outputs
- **Fallback content** (`src/data/fallbackCases.js`, `fallbackEndings.js`): If LLM fails or returns invalid JSON, pre-written content is used
- **Retry logic** (`src/api/ai.js:193-216`): Up to 3 attempts with backoff for API failures
- **JSON robustness** (`src/api/ai.js:7-98`): 5-level JSON extraction strategy (direct parse → outermost brace → case-block extraction → endings extraction → error)
- **Rule engine fixpoint** (`src/expert/inferenceEngine.js:83`): MAX_ITERATIONS = 100 prevents infinite loops

### Performance Characteristics
- **API calls per playthrough**: Exactly 2 (Cases 9-10 generation, then endings generation) — `README.md:227-235`
- **Expert system**: Fires in < 1ms (40 rules, max 100 iterations, pure computation)
- **LLM latency**: 2–4 seconds per call (noted in `README.md:332`)

## 8. LIMITATIONS & FUTURE WORK

### Current Limitations (from `README.md:329-335`)
1. **JSON reliability**: ~1% of OpenRouter responses return malformed JSON; retry system adds 2–4s latency.
2. **Narrative variance**: Llama 3.1 occasionally produces less literary prose; tone consistency ~95%.
3. **No offline mode**: Cases 9-10 and endings require active internet connection.
4. **Mobile UX**: Severity slider is finicky with touch input on mobile.
5. **No automated tests**: Only manual playtesting recommended.
6. **No `archetypes.js` file**: README references `src/data/archetypes.js` (line 105) but this file does not exist — archetype definitions are embedded in localization files.

### Suggested Future Work
1. **Add comprehensive testing**: Unit tests for the expert system (rule firing, working memory construction, conclusion selection), integration tests for the profile tracker, and end-to-end tests for the game flow. This is the most impactful improvement for an academic project.
2. **Expand the expert system**: Add backward-chaining capability (goal-directed reasoning), support for rule chaining across categories, and a confidence-weighted conflict resolution strategy beyond simple priority ordering.
3. **Multiplayer / analytics**: The README roadmap (`README.md:429-439`) lists multiplayer archetype comparison, an analytics dashboard (anonymised archetype distribution), and a custom scenario editor.

- **Node.js 18+** and npm
- **OpenRouter API key** (free tier available at [openrouter.ai](https://openrouter.ai) — no credit card required)

### Steps

1. **Clone the repository**
   ```bash
   git clone https://github.com/yourusername/the-tribunal.git
   cd the-tribunal
   ```

2. **Install dependencies**
   ```bash
   npm install
   ```

3. **Create `.env` file**
   ```bash
   cp .env.example .env
   ```

4. **Add your OpenRouter API key** to `.env`:
   ```
   VITE_OPENROUTER_API_KEY=your_key_here
   ```

5. **Start development server**
   ```bash
   npm run dev
   ```

6. **Open in browser**
   ```
   http://localhost:5173
   ```

**Key academic insight** (from `README.md:402`): *"The AI does not evaluate the player, it observes their patterns and mirrors them back."* The expert system is a reasoning layer that sits between raw player data and narrative output — it classifies, recommends, and explains, while the LLM provides only the literary prose layer on top of structurally predetermined outcomes.
